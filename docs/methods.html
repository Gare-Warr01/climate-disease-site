Overview of the Methods

This project was built to be transparent, reproducible, and easy to follow.
Every graph, table, and comparison on this website comes from a clear, step-by-step workflow in R that someone else could recreate.

Workflow at a Glance

Most of the analysis on this site follows the same basic pipeline: 

Raw data (NOAA, CDC) 
    ↓
Cleaning & standardizing in R (tidyverse)
    ↓
Joining climate + disease by year/state
    ↓
Summarizing & visualizing (ggplot2, tables)
    ↓
Interpreting patterns & limits

The goal is not to build the most complex model possible, but to use a clear, repeatable process that makes the climate–disease story visible and honest.

Data Sources

To explore how climate conditions relate to mosquito-borne disease, this project draws from two major groups of datasets.

Climate Data

  • Sourced from NOAA climate summaries.
  • Includes temperature, precipitation, and drought indices such as PDSI.
  • Aggregated to the state–year level for consistent national comparison.

Disease Data

  • Built from human case data for vector-borne diseases using CDC public health surveillance.
  • Focuses on four main diseases:
    • West Nile virus (WNV)
    • Lyme disease
    • Dengue
    • Malaria
  • Includes reported cases per state and year.
  • Emphasizes long-term, national patterns rather than individual outbreaks.

All datasets were used strictly for educational purposes in GEOG 3000 – Climate Change in Science & Society.

Data Cleaning & Preparation

All data cleaning was performed in R using the tidyverse.

Key steps included:

  • Importing CSVs with readr::read_csv()
  • Standardizing column names for readability
  • Filtering to valid states and meaningful years
  • Checking for missing or inconsistent values
  • Removing invalid observations (e.g., negative case counts)
  • Creating tidy, analysis-ready tables

One of the biggest lessons here: real-world data is imperfect, and careful cleaning is essential for honest analysis.

Combining Climate & Disease Data

To make meaningful comparisons, the datasets needed to align.

This involved:

  • Standardizing state names / abbreviations
  • Ensuring matching year formats across files
  • Creating a shared state–year joining key
  • Using dplyr::left_join() to merge climate indicators with disease case counts

The unified dataset supports the visuals throughout:

  • Climate Trends
  • Disease Trends
  • Climate–Disease Comparison
  • Utah Focus

The goal was not prediction, but to create a clean structure where visual patterns could be explored with confidence.

One Example R Workflow

This short example shows the kind of R pipeline used across the project: load data, clean it, join climate and disease, and calculate simple summaries.

Show example R code: load, clean, join, summarize 
library(tidyverse)

# 1. Load yearly climate and disease data
climate_temp <- readr::read_csv("data/climate_year_temp.csv") |>
  mutate(year = as.integer(year))

master_disease <- readr::read_csv("data/master_disease_comparison.csv") |>
  mutate(
    year  = as.integer(year),
    cases = as.numeric(cases)
  )

# 2. Filter to one disease (West Nile) and national totals
westnile_us <- master_disease |>
  filter(level == "national", disease == "West Nile") |>
  select(year, cases)

# 3. Join climate and disease by year
climate_westnile <- westnile_us |>
  left_join(climate_temp, by = "year")

# 4. Create a simple summary table
climate_westnile |>
  summarize(
    years_in_study   = n(),
    mean_temp_anom   = mean(temp_anomaly, na.rm = TRUE),
    mean_wnv_cases   = mean(cases, na.rm = TRUE)
  )

The graphs and models shown on other tabs are built by extending this same pattern:
read → clean → join → summarize → visualize.

Visualization & Analysis in R

All graphs were created using ggplot2, chosen for its clear, consistent visual style.

The project uses:

  • Line charts for long-term climate and disease trends
  • Faceted plots for comparing regions, states, or diseases
  • Scatterplots with trend lines to explore simple climate–disease relationships
  • Clean, colorblind-friendly palettes for accessibility

This project emphasizes:

  • Visual storytelling
  • Exploratory data analysis
  • Clear communication

Readers should be able to interpret trends without needing advanced statistical training. For extra support, the How to Read These Graphs page walks through the main plot types step by step.

Reproducibility & Workflow

A major priority was ensuring this project was reproducible.

This was supported by:

  • Building each page in R Markdown
  • Keeping explanations and code together in the same document
  • Using self-updating graphics that knit directly from the underlying data
  • Hosting the entire project on GitHub Pages
  • Maintaining a clear, folder-based file structure for data and code

Reproducibility is not just a coding goal — it’s a commitment to scientific clarity and transparency.

Analytical Choices & Limitations

This project uses descriptive and visual analysis rather than complex predictive modeling.
This choice keeps the results accessible and avoids overstating what the datasets can support.

Limitations include:

  • Differences in reporting practices between states
  • Possible under-reporting of human vector-borne disease cases
  • Occasional gaps or inconsistencies in climate data
  • State-level aggregation that cannot capture city-level or neighborhood variation

These limitations do not prevent meaningful exploration — they simply frame how the trends should be interpreted. Readers should treat the results as patterns to think about, not as final answers.

Software & Versioning

To support reproducibility, all work was completed using:

  • R (version 4.x)
  • RStudio as the development environment
  • tidyverse for data cleaning and manipulation
  • ggplot2 for visualization
  • knitr and rmarkdown for rendering the website

Using consistent software versions helps ensure that graphs and results remain stable over time and can be reproduced by others following the same steps.
